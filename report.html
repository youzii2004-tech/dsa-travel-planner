IIntroduction

Navigation is one of the oldest and most fundamental problems in applied mathematics. Finding the fastest route between two cities, the least costly connection in a network, or the minimum path in any weighted graph underpins transportation logistics, internet routing protocols, social network analysis, and GPS systems worldwide. At the core of nearly every modern pathfinding system lies a single elegant algorithm: Dijkstra's Shortest Path Algorithm, proposed by Dutch computer scientist Edsger W. Dijkstra in 1956 and published in 1959.

This project implements a fully interactive Travel Route Planner rendered on a real map of the United States. Ten major American cities are connected by a weighted graph where edge weights represent driving distances in miles. A user selects departure and destination cities, and watches Dijkstra's algorithm animate through the graph in real time — visiting nodes, relaxing edges, and highlighting the optimal route. The map is built with Leaflet.js on actual geographic coordinates, grounding the algorithm in the real world.

IIProblem Statement

Given a weighted undirected graph G = (V, E) where V is a set of cities and each edge e ∈ E carries a positive weight representing road distance in miles, find the path from source city s to target city t such that the total edge weight along the path is minimised. Output the ordered sequence of cities and the total distance.

This is the Single-Source Shortest Path (SSSP) problem. Brute-force enumeration of all paths has exponential complexity and is infeasible even for small graphs. Dijkstra's algorithm solves this optimally in polynomial time, with the essential precondition that all edge weights are non-negative — satisfied here because distances are positive mile values.

IIIObjectives
  • Model a network of 10 US cities as a weighted undirected graph with 18 edges
  • Implement Dijkstra's algorithm using a min-heap priority queue for O((V+E) log V) performance
  • Animate every step of the algorithm on a real geographic map — node visits, edge relaxations, path highlighting
  • Display the optimal route with total distance and a complete execution log
  • Provide animation speed controls and a full reset function
IVData Structures Used

Adjacency List — Graph Representation

The graph is stored as a hash map from city name to an array of {neighbor, weight} pairs. Adjacency lists are preferred over matrices here because the graph is sparse: with 10 nodes and 18 edges, a matrix would be 100 cells with 82 empty. The list stores only existing connections in O(V + E) space.

Min-Heap — Priority Queue

Dijkstra's algorithm repeatedly extracts the unvisited node with the minimum known distance. A min-heap supports both insert and extract-minimum in O(log n) time, giving the algorithm its O((V+E) log V) overall complexity. This is the critical data structure that separates Dijkstra's from a naive O(V²) scan-based approach.

Distance and Predecessor Maps

Two hash maps track algorithm state: dist[v] stores the shortest known distance from source to v (initialised ∞ for all but source), and prev[v] stores the predecessor of v on the current best path. After termination, the optimal path is reconstructed by walking backwards through prev[] from the destination to the source.

VDijkstra's Algorithm

Dijkstra's is a greedy algorithm. Its correctness rests on the shortest-subpath property: any subpath of a shortest path is itself a shortest path. This allows the algorithm to permanently commit to a shortest distance for a node the moment it is extracted from the priority queue — because no shorter path can be discovered later without traversing a node that already has a known shorter distance.

Edge Relaxation

The central operation is relaxation. For a current node u and its neighbor v connected by edge weight w: if d[u] + w < d[v], the distance table is updated (d[v] ← d[u] + w), the predecessor is recorded (prev[v] ← u), and v is re-inserted into the priority queue. This "relaxes" the upper bound on d[v] toward its true minimum.

Why greedy works here: Once a node u is extracted from the min-heap with distance d[u], no future path can improve upon it — any such path would start from a node with an equal or greater distance (since the heap always gives us the smallest). All 18 edges in this graph carry positive mile values, so this precondition is fully satisfied.

VIIPseudocode
function DIJKSTRA(graph, source, target):
    // Initialise all distances to infinity
    for each city v in graph:
        dist[v] ← ∞;   prev[v] ← null
    dist[source] ← 0

    pq ← MinHeap()
    pq.insert( (0, source) )
    seen ← empty set

    while pq not empty:
        (d, u) ← pq.extractMin()
        if u in seen: continue   // skip stale entries
        seen.add(u)
        if u == target: break

        for each (neighbor v, weight w) in graph[u]:
            if v in seen: continue
            nd ← dist[u] + w
            if nd < dist[v]:          // edge relaxation
                dist[v] ← nd
                prev[v] ← u
                pq.insert( (nd, v) )

    // Reconstruct path via prev[] backtracking
    path ← [];   cur ← target
    while cur ≠ null:
        path.prepend(cur);   cur ← prev[cur]
    return path, dist[target]
VIIIImplementation Details

Real Geographic Map — Leaflet.js

The application uses Leaflet.js with CartoDB Dark Matter tiles on OpenStreetMap infrastructure. All 10 cities are placed at real latitude/longitude coordinates — Seattle at (47.61°N, 122.33°W), Miami at (25.76°N, 80.19°W), and so on. This replaces an earlier SVG approach that suffered a fatal coordinate scaling bug: a constant S = 8.3 multiplied screen-space pixel values producing coordinates like 5,395px — far outside an 820px viewbox, rendering all nodes invisible.

Node State Animation

Each city marker cycles through visual states: default (steel), source (green), destination (red), visiting (cyan), explored (grey), and path (amber). Timing is configurable via the Slow / Normal / Fast selector, implemented using async/await sleep calls between each algorithm step.

Execution Log Panel

Every algorithm event — initialization, node visits with current distances, edge relaxations with before/after values, and final path confirmation — is logged in a scrollable panel below the map. Entries are numbered and colour-coded: amber for updates, cyan for visits, green for the final path.

IXTime & Space Complexity
Graph constructionO(V + E)
Min-heap insertO(log V)
Min-heap extract-minimumO(log V)
Dijkstra's total — binary heapO((V + E) log V)
Path reconstructionO(k) — path length
Space — dist[] + prev[] tablesO(V)
Space — priority queue (worst)O(E)
Space — adjacency listO(V + E)

For V = 10 cities and E = 18 edges: O((10 + 18) × log 10) ≈ 90 operations — computed in microseconds. The animation delay is intentional, allowing the user to follow each step. Real-world GPS systems run the same algorithm on graphs of millions of nodes, demonstrating its scalability.

XIConclusion

This project delivers a complete, animated, and visually compelling implementation of Dijkstra's shortest path algorithm on a real geographic map. The algorithm correctly identifies optimal routes for all 45 possible city pairings within the 10-city US network, verified against known highway distances. The Leaflet.js integration with CartoDB tiles provides a professional map context that bridges the gap between abstract graph theory and real-world navigation.

Working on this project solidified my understanding of greedy algorithms, the mathematical guarantee behind the extract-minimum strategy, and the critical performance role of the min-heap priority queue. It also reinforced the engineering principle of separating algorithm logic from rendering logic — a lesson learned by debugging the coordinate scaling failure in the initial SVG-based implementation.
